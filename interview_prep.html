Inference Code — Exact Logic

ml-server/app.py
@app.route("/api/ml/analyze-image", methods=["POST"])
def analyze_image():
    data = request.json
    img_str = data.get("image", "")

    # 1. Strip data URI prefix if present
    if "base64," in img_str:
        img_str = img_str.split("base64,")[1]

    # 2. Decode base64 → PIL Image
    img_data = base64.b64decode(img_str)
    img = Image.open(io.BytesIO(img_data))

    # 3. Run YOLOv8 inference
    results = yolo_model(img)

    # 4. Extract bounding box class names + confidence scores
    detections = []
    for r in results:
        for box in r.boxes:
            cls_id  = int(box.cls[0].item())
            conf    = float(box.conf[0].item())
            cls_name = yolo_model.names[cls_id]
            detections.append({"class": cls_name, "confidence": round(conf, 2)})

    unique_classes = list(set([d['class'] for d in detections]))
    return jsonify({"detections": detections, "summary": ", ".join(unique_classes)})

What is YOLOv8?

YOLO = You Only Look Once. A single-stage real-time object detection architecture.

  • ✦Divides image into a grid; each cell predicts bounding boxes + class probabilities simultaneously
  • ✦Single forward pass — much faster than two-stage detectors (R-CNN family)
  • ✦YOLOv8 (Ultralytics) is the state-of-the-art version — anchor-free, better small object detection
  • ✦Output: bounding box coords (x, y, w, h) + class label + confidence score per detected object

Training Details

  • ✦Model: best (1).pt — custom trained weights (22MB)
  • ✦Task: object detection on garbage/waste classes
  • ✦Framework: ultralytics Python package
  • ✦Training done in EPICS.ipynb notebook
  • ✦Input: photos uploaded by citizens as part of complaint filing
  • ✦Output: list of detected garbage types with confidence % per detection
🔑
Key interview answer — "why YOLO and not a CNN classifier?" "A simple CNN image classifier would only tell us is there garbage? (yes/no). YOLOv8 gives us where the garbage is (bounding boxes) and how many objects are detected — which is more useful for a municipal officer assessing severity. Multiple detections with high confidence = higher priority complaint."

Inference Code — Exact Logic

ml-server/app.py
# Models loaded once at startup (global scope)
vectorizer = joblib.load(os.path.join(MODEL_DIR, 'tfidf_vectorizer.pkl'))
cat_model  = joblib.load(os.path.join(MODEL_DIR, 'category_model.pkl'))
sev_model  = joblib.load(os.path.join(MODEL_DIR, 'severity_model.pkl'))

@app.route("/api/ml/analyze-text", methods=["POST"])
def analyze_text():
    text = request.json.get("text", "")
    if not text.strip():
        return jsonify({"category": "", "severity": ""})

    # 1. Vectorize text using fitted TF-IDF
    vec = vectorizer.transform([text])   # sparse matrix (1, n_features)

    # 2. Two separate classifiers share same vectorized input
    cat = cat_model.predict(vec)[0]    # e.g. "Water", "Roads", "Sanitation"
    sev = sev_model.predict(vec)[0]    # e.g. "High", "Medium", "Low"

    return jsonify({"category": cat, "severity": sev})

What is TF-IDF?

TF-IDF = Term Frequency × Inverse Document Frequency. Converts text into a numerical vector.

  • ✦TF — how often a word appears in this document
  • ✦IDF — penalizes words that appear in many documents (common words like "the", "is" get low weight)
  • ✦Result: unique, informative words like "pothole", "sewage", "leakage" get high scores
  • ✦vectorizer.transform() at inference — uses already-fitted vocabulary, no retraining

Model Artifacts

  • ✦tfidf_vectorizer.pkl — 25KB, fitted vocabulary + IDF weights
  • ✦category_model.pkl — 26KB, classifies into department categories (Water, Roads, Sanitation, Electricity, etc.)
  • ✦severity_model.pkl — 2MB (larger = more complex model), classifies into High / Medium / Low
  • ✦Serialized with joblib (preferred over pickle for sklearn — better compression for large arrays)
  • ✦Training done in nlp.ipynb notebook
💡
Why two separate models (category + severity) instead of one multi-output model? "Category and severity are two independent classification tasks with different label spaces. Training them separately keeps each model focused — the severity model can be a more complex algorithm (hence its 2MB size vs 26KB category model) without affecting category performance. It also makes them independently replaceable."

Model Loading Strategy — Startup vs Per-Request

All four models are loaded once at startup in global scope — not on every request. This is critical for performance:

app.py — startup loading
# Loaded ONCE when Flask starts — global scope
MODEL_DIR  = os.path.join(os.path.dirname(os.path.dirname(__file__)), "ML Model")
vectorizer = joblib.load(os.path.join(MODEL_DIR, 'tfidf_vectorizer.pkl'))
cat_model  = joblib.load(os.path.join(MODEL_DIR, 'category_model.pkl'))
sev_model  = joblib.load(os.path.join(MODEL_DIR, 'severity_model.pkl'))
yolo_model = YOLO(os.path.join(MODEL_DIR, 'best (1).pt'))

# At inference — just call .transform() and .predict(), no reloading
⚡
Why not load per-request? joblib.load and YOLO weight loading takes 1–5 seconds. Loading on every request would make the app unusable. Global scope = loaded once, inference is milliseconds.

General Interview Questions
Project framing, your role, and architecture decisions
Q01 Tell me about IMPACT and your role in it. ▼

"IMPACT is a full-stack civic complaint management platform we built as a group EPICS project. It digitizes the entire municipal grievance lifecycle — citizens can file complaints with photo evidence, track resolution status, pay traffic challans, and view ward info with Google Maps integration. There are three user roles: citizens, municipal officers, and traffic police.

My contribution was the entire AI/ML layer. I trained a YOLOv8 model to detect garbage and waste objects in complaint photos, and built TF-IDF NLP classifiers that read the complaint text and automatically predict the complaint category (Water, Roads, Sanitation, etc.) and severity (High/Medium/Low). Both models are served through a Flask microservice I built that runs on port 8000 and is automatically spawned by our Node.js backend."

Q02 Why did you use a separate Flask server instead of running ML inside Node.js? ▼

There are fundamental reasons why Python ML can't run inside Node.js:

  • Ecosystem mismatch: YOLOv8 (ultralytics) and scikit-learn are Python-only libraries. There's no JS equivalent with the same maturity, model compatibility, or performance.
  • Process isolation: Running ML in a separate process means a crash in inference doesn't bring down the entire Node.js API server.
  • Scalability: The ML server can be independently scaled or replaced — the Node.js backend only needs to know the API contract (POST endpoint + JSON schema).
  • Auto-spawn convenience: We used Node's spawn() to start the Flask server automatically, so there's zero extra setup for anyone running the project.
Q03 How does the auto-spawn of the Flask server work? ▼

In server/server.js, when the Node.js server starts, it runs a spawn() call with the path to the Python venv and app.py:

  • It locates the virtual environment's Python binary (cross-platform — Windows/Mac paths differ)
  • Spawns python app.py with the ml-server/ directory as the working directory
  • The Flask server starts on port 8000 in the background
  • No developer needs to open a separate terminal — npm run dev starts everything: React (5173) + Express (5000) + Flask (8000)
  • This is the "Dev Experience" win — it makes the ML server feel like a native part of the Node stack
Q04 What would you improve in the ML layer if you had more time? ▼

Focusing on the ML side specifically:

  • Confidence threshold for YOLO: Right now all detections are returned regardless of confidence. I'd add a threshold (e.g., conf > 0.5) to filter out low-confidence noisy detections before returning to the user.
  • Return bounding box coordinates: The current response only returns class names and confidence, not the actual pixel coordinates. Returning coords would let the frontend draw boxes on the image.
  • Severity model improvement: The severity model is 2MB (much larger than the 26KB category model) — suggesting it's more complex. I'd look at whether a lighter model (Logistic Regression + better features) achieves similar accuracy.
  • Active learning pipeline: As officers correct AI-predicted categories, those corrections could feed back into retraining — improving accuracy over time without manual labeling.
  • Async processing: Currently the ML call is synchronous in the complaint filing flow. I'd make it async so the complaint saves immediately and AI results update afterward.

YOLOv8 Q&A
Computer vision deep-dive questions
Q05 Explain how YOLO works. Why is it called "You Only Look Once"? ▼

Traditional object detectors like R-CNN are two-stage: first propose regions of interest, then classify each region. YOLO is a single-stage detector:

  • Grid division: The image is divided into an S×S grid
  • Single forward pass: Each grid cell simultaneously predicts bounding box coordinates (x, y, w, h), objectness score, and class probabilities — all in ONE pass through the neural network
  • "Only Once": You look at the image once — one forward pass gives you all detections, instead of running the classifier hundreds of times on proposed regions
  • YOLOv8 specifically: Anchor-free (no predefined box shapes), uses a CSP backbone, and has separate detection heads per scale for better small object detection
  • Speed vs accuracy: YOLO trades some accuracy for massive speed gains — ideal for real-time applications like our complaint photo analysis
Q06 What is the .pt file? What does it contain? ▼
  • .pt = PyTorch model file — the standard format for saving PyTorch neural network weights
  • Contents: The trained neural network weights (all the learned parameters) + model architecture metadata
  • best (1).pt — "best" is the convention YOLOv8 training uses for the checkpoint with the best validation mAP (mean Average Precision). It saves the best weights encountered during training, not just the last epoch's weights.
  • Size — 22MB: This is a relatively small YOLO model — likely YOLOv8n (nano) or YOLOv8s (small) variant, optimized for speed over maximum accuracy. Appropriate for a web application where inference latency matters.
  • Loading: YOLO("best (1).pt") from the ultralytics library loads the weights into memory ready for inference
Q07 What is mAP and how would you evaluate your YOLOv8 model? ▼

mAP = mean Average Precision — the standard metric for object detection models.

  • Precision: Of all boxes the model predicted as garbage, what fraction actually were garbage?
  • Recall: Of all actual garbage in images, what fraction did the model find?
  • IoU (Intersection over Union): Measures how well the predicted bounding box overlaps the ground truth box. A prediction counts as correct if IoU > threshold (e.g., 0.5)
  • AP per class: Area under the Precision-Recall curve for each class
  • mAP@0.5: Mean AP at IoU threshold 0.5 — the most common reported metric
  • For our model: YOLOv8 training automatically generates a results.csv with mAP@0.5 and mAP@0.5:0.95 per epoch
Q08 How did you handle the base64 image input? Why base64? ▼
  • Why base64: JSON can only transmit text. Images are binary data. base64 encodes binary as ASCII text, making it embeddable in a JSON payload. The frontend sends the photo as a base64 string inside the JSON body.
  • Data URI prefix: When a browser reads a file with FileReader.readAsDataURL(), it prepends data:image/jpeg;base64,. My code strips this prefix before decoding: img_str.split("base64,")[1]
  • Decoding pipeline: base64.b64decode() → bytes → io.BytesIO() → PIL.Image.open() → PIL Image object → YOLO input
  • Trade-off: base64 increases payload size by ~33% vs sending binary. For a civic complaint photo, this is acceptable. For high-volume production, you'd use multipart/form-data instead.

NLP Q&A
Text classification deep-dive questions
Q09 Explain TF-IDF. Why did you use it over word embeddings like Word2Vec? ▼

TF-IDF formula: TF-IDF(word) = TF(word in doc) × log(N / df(word))

  • TF (Term Frequency): count of word in this document / total words in document
  • IDF (Inverse Document Frequency): log(total documents / documents containing this word). Rare words get high IDF — they're more informative
  • Result: Words like "pothole", "sewage", "electricity" score high for their respective complaint types; "the", "is", "a" score near zero

Why TF-IDF over Word2Vec/BERT for this use case:

  • Domain is keyword-driven: Municipal complaints have highly distinctive keywords per category. "pothole" → Roads, "leakage" → Water, "garbage" → Sanitation. TF-IDF captures this perfectly.
  • Training data size: Word2Vec needs large corpora. For a domain-specific civic complaints dataset, TF-IDF + a simple classifier is more reliable with limited labeled data.
  • Speed: TF-IDF vectorization is near-instant at inference. BERT would add 200–500ms per request.
  • Interpretability: You can inspect which words have highest TF-IDF scores per category — useful for debugging and explaining predictions.
Q10 What classifier did you use for category and severity? Why? ▼

The exact classifier can be inferred from the pkl file sizes:

  • category_model.pkl (26KB) — very small, suggesting a lightweight model like Logistic Regression, Naive Bayes, or Linear SVC
  • severity_model.pkl (2MB) — much larger, suggesting a more complex model like Random Forest, Gradient Boosting, or SVC with RBF kernel

If asked which specifically, be honest: "The category classifier is a lightweight linear model given its small size — likely Logistic Regression or Naive Bayes, both of which work excellently with TF-IDF sparse vectors. The severity model is more complex given its 2MB footprint, possibly an ensemble or SVC with a non-linear kernel."

General justification for linear models with TF-IDF:

  • TF-IDF produces high-dimensional sparse vectors. Linear models (LR, Naive Bayes) are very effective in high-dimensional sparse feature spaces — this is well-established in NLP research
  • Fast training, fast inference, low memory during inference
Q11 Why use joblib instead of pickle for the NLP models? ▼
  • joblib is optimized for NumPy arrays: sklearn models internally use large NumPy arrays (coefficient matrices, vocabulary arrays). joblib serializes these much more efficiently than pickle using memory-mapped files and compression.
  • sklearn recommendation: The official sklearn documentation recommends joblib over pickle for sklearn estimators.
  • Speed: joblib.load is faster for large numpy-backed objects — relevant for the 2MB severity model.
  • In contrast: HealBridge AI used pickle directly — both work, but joblib is the better practice for sklearn artifacts.
Q12 How would you handle a complaint text in a language other than English? ▼

This is a genuine limitation of the current system — a great future improvement to mention:

  • Current behavior: The TF-IDF vectorizer was fitted on English text. Non-English input would be vectorized using the English vocabulary — most tokens would be OOV (out-of-vocabulary) resulting in a near-zero vector and a random/wrong prediction.
  • Fix 1 — Translation preprocessing: Add a Google Translate API call before vectorization to translate to English first.
  • Fix 2 — Multilingual TF-IDF: Retrain the TF-IDF + classifiers on a multilingual complaints dataset covering Hindi, Marathi, etc.
  • Fix 3 — Multilingual BERT (mBERT): Replace TF-IDF with a multilingual transformer embedding — works across 100+ languages without translation.
  • The README mentions multilingual support as a planned feature — reference this directly.
